Storybook for TanStack React is Storybook's [framework](../../contribute/framework.mdx) integration for [TanStack Router](https://tanstack.com/router) and [TanStack Start](https://tanstack.com/start/latest) applications built with [React](https://react.dev/) and [Vite](https://vitejs.dev/). It builds on `@storybook/react-vite` and adds router-aware story rendering, automatic router mocking, and TanStack Start stubs so components that depend on routing or server functions can render inside Storybook without booting your full app runtime.

If you are building a plain React and Vite application without TanStack Router, use [`@storybook/react-vite`](./react-vite.mdx) instead.

## Configure

Storybook for TanStack React uses Vite through [`@storybook/builder-vite`](../../builders/vite.mdx) and automatically wraps each story in a memory-backed TanStack Router. This gives you a working router context in Storybook without having to boot your full application shell.

Under the hood, the integration also intercepts TanStack runtime modules so Storybook behaves more like a TanStack Router or TanStack Start app during development.

Out of the box, it supports these workflows:

* Pass a TanStack Route directly as a story `component`
* Set the initial route, params, and query string per story
* Override `loader`, `beforeLoad`, and more on the story route without modifying the original route object via `routeOverrides`
* Automatically mock `@tanstack/react-router` so navigation hooks work in stories and navigation attempts can be observed
* Automatically stub TanStack Start server/runtime entry points so components using server functions can render in Storybook
* Use typed preview helpers through `definePreview()` and `preview.meta()`

### Overriding route options per story

When a route has a `loader` or `beforeLoad` that calls real APIs, you can override those options per story without modifying the original route object. Pass `routeOverrides` under `parameters.tanstack.router`. Each key is a route ID and the value can override `loader`, `beforeLoad`, `validateSearch`, `loaderDeps`, and `context`.

Use `'__root__'` as the key to target the root route.

### Automatic TanStack Router and Start mocks

When this integration runs in Storybook, it automatically redirects `@tanstack/react-router` imports to a Storybook-compatible mock layer. That mock re-exports TanStack Router APIs, keeps hooks such as `useNavigate()`, `useSearch()`, and `useParams()` available in stories, and wires navigation attempts into Storybook spies.

For TanStack Start apps, the integration also stubs TanStack Start server and runtime entry points. This is what allows components that depend on server functions or Start-specific runtime modules to render in Storybook without a running Start server.

In practice, this means you can usually render TanStack Start components directly, and `createServerFn()` handlers are replaced with mock functions that you can observe and override in stories and tests.

### Handling server-only dependencies

TanStack Start apps often import server-only packages (e.g. database clients, auth libraries) at module scope inside route files. When Storybook loads the route tree, those imports can crash the browser. The integration handles this at three layers:

#### Layer 1 — Framework-level mocks (automatic)

The preset already intercepts `@tanstack/react-start`, `@tanstack/react-start/server`, `@tanstack/start-storage-context`, and related TanStack modules. It also replaces `createServerFn()` handlers with mock functions. You do not need to do anything for these.

#### Layer 2 — App-level server modules

When your routes import app-specific server code (e.g. `~/db/client`, `~/auth/index.server`), use Storybook's mocking with a `__mocks__` file to prevent the real module — and its Node.js dependencies — from loading in the browser.

**Step 1**: Register the mock in `.storybook/preview.ts`:

<CodeSnippets path="tanstack-react-mock-module-preview.md" />

**Step 2**: Create `src/db/__mocks__/client.ts` next to the real module. Use only `import type` so no server packages are pulled in:

<CodeSnippets path="tanstack-react-mock-db-client.md" />

<Callout variant="info">

**Why a `__mocks__` file instead of automocking?** Storybook's automocking (`sb.mock(import(...))` without a `__mocks__` file) replaces *functions* but still **evaluates the original module** and its imports. For modules that import `postgres`, `pg`, or other Node.js-only packages, the original module must never be evaluated. A `__mocks__` file is the only approach that completely prevents evaluation of the original module and its dependency chain.

</Callout>

#### Identifying what to mock

If Storybook throws a browser error like `does not provide an export named 'default'` or `AsyncLocalStorage is not defined`, open the error stack trace and find the deepest module that imports a Node.js API. That module is what needs to be mocked at its point of entry.

Common patterns in TanStack Start apps:

| Import | Root cause | Solution |
| --- | --- | --- |
| `~/db/client` | imports `postgres` | `__mocks__` file (Layer 2) |
| `~/auth/index.server` | imports `~/db/client` | already covered if `db/client` is mocked |
| `~/utils/*.functions` | imports `createServerFn` | already handled by the preset (Layer 1) |

### Story fails to render with an error about modules not providing a default export

This usually means a server-only module is being imported in the browser. Check the error stack trace to find the module and add a Storybook mock for it as described in [Handling server-only dependencies](#handling-server-only-dependencies).

### When should I use `@storybook/tanstack-react` instead of `@storybook/react-vite`?

Use `@storybook/tanstack-react` when your components rely on TanStack Router or TanStack Start APIs and you want Storybook to provide router context, typed route parameters, automatic router mocking, and mocked TanStack Start server-function behavior.

Use [`@storybook/react-vite`](./react-vite.mdx) when your app is a standard React and Vite project without TanStack Router.

### How do I write a story for a route with dynamic params (e.g. `/$id`)?

Import the route object and supply `params` alongside `routeOverrides` under `parameters.tanstack.router`. The `params` object is interpolated into the URL, and `routeOverrides` lets you stub the loader without touching the original route.

```tsx
import { Route } from './$id'

const meta = {
  parameters: {
    tanstack: {
      router: {
        route: Route,
        params: { id: '42' },
        routeOverrides: {
          '/showcase/$id': {
            loader: () => ({ item: mockItem }),
          },
        },
      },
    },
  },
} satisfies Meta<typeof Route>
```

For the full set of options, see [`RouterParameters`](#parameters).

### My route loader calls `queryClient.ensureQueryData(...)` and it fails with "data is undefined"

**Why does this happen?** The route loader runs inside the Storybook router and calls your real query function. When that function is a `createServerFn` stub (a no-op), it returns nothing, so the QueryClient cache is empty and `ensureQueryData` finds no data.

**Fix:** Pre-seed the QueryClient in a story-level `loaders` hook so the cache is warm before the loader runs:

```tsx
loaders: [
  async ({ parameters }) => {
    const qc: QueryClient = parameters.tanstack?.router?.context?.queryClient
    qc.setQueryData(['items', 'detail', itemId], { item: mockItem })
  },
],
```

The QueryClient instance must be provided via `parameters.tanstack.router.context` **and** wrapped with a `QueryClientProvider` decorator so the same instance is available to both the loader and the component tree.

### How do I mock server functions (`createServerFn`) in stories?

The framework stubs TanStack Start entry points automatically (see [Automatic TanStack Router and Start mocks](#automatic-tanstack-router-and-start-mocks)), so most `createServerFn` calls are already replaced with mock functions you can override per story. For app-level server modules not covered by the preset, add a Vite plugin in `viteFinal` that replaces server files with no-op proxy exports:

```ts
// .storybook/main.ts → viteFinal
config.plugins.push({
  name: 'storybook:stub-server-files',
  enforce: 'pre',
  transform: {
    filter: { id: /\.server\.[mc]?[jt]sx?$/ },
    handler(code, id) {
      if (id.includes('node_modules')) return null
      // Extract named exports and re-export them as callable proxies
      const names = /* ...parse named exports from `code`... */
      const lines = [
        `const _noop = () => {}`,
        `const _proxy = new Proxy(_noop, { get: () => _proxy, apply: () => _proxy })`,
      ]
      for (const name of names) lines.push(`export const ${name} = _proxy`)
      return { code: lines.join('\n'), map: null }
    },
  },
})
```

This replaces every named export with a callable proxy so imports from server files never crash in the browser.

### I get `Buffer is not defined` or `getRequest is not found`

**Why does this happen?** A component import chain reaches Node-only code (for example an auth utility that imports `postgres`). The browser does not have these built-ins.

**Fix:** Either stub the entire module using a `__mocks__` file (see [Handling server-only dependencies](#handling-server-only-dependencies)), or alias specific packages to empty mocks in `viteFinal`:

```ts
config.resolve.alias = {
  ...config.resolve.alias,
  postgres: path.resolve(__dirname, '../__mocks__/postgres.ts'),
}
```

### How do I set search params or a hash in a route story?

Use `path` for a URL hash and `query` for search params under `parameters.tanstack.router`:

```tsx
export const WithHash: Story = {
  parameters: {
    tanstack: {
      router: { path: '/#section' },
    },
  },
}

export const WithSearch: Story = {
  parameters: {
    tanstack: {
      router: { query: { tab: 'details', page: '2' } },
    },
  },
}
```

### Where do I import doc blocks (`Meta`, `Canvas`, `Controls`) in MDX pages?

In Storybook 10, import doc blocks from `@storybook/addon-docs/blocks`:

```mdx
import { Meta, Canvas, Controls } from '@storybook/addon-docs/blocks'
```

Do not import from `@storybook/blocks` (used in Storybook 8/9) or `storybook/blocks` (not a valid export in Storybook 10).

### How do I provide React context providers (theme, toast, auth) to all stories?

Add decorators in `.storybook/preview.ts` for providers that every story needs, or apply them per story file for narrower scope:

```tsx
// .storybook/preview.ts — global providers
const preview = definePreview({
  decorators: [
    (Story) =>
      React.createElement(QueryClientProvider, { client: queryClient },
        React.createElement(Story, null),
      ),
  ],
})
```

```tsx
// In a story file — additional providers scoped to that file
decorators: [
  (Story: React.ComponentType) => (
    <LoginModalProvider>
      <ToastProvider>
        <Story />
      </ToastProvider>
    </LoginModalProvider>
  ),
],
```

### My styles are missing in Storybook

Import your application CSS in `.storybook/preview.ts` so it is bundled with the preview:

```ts
import '../src/styles/app.css'
```

## API

### Parameters

This framework contributes the following [parameters](../../writing-stories/parameters.mdx) to Storybook under the `tanstack.router` namespace:

| Parameter | Type | Description |
| --- | --- | --- |
| `route` | `AnyRoute \| route options object` | Supplies a route instance directly or creates a temporary story route from route options. Storybook extracts the route's React component automatically when no `component` is set on the meta. |
| `path` | `string` | Sets the initial URL path for the story router. |
| `params` | `Record<string, string>` | Interpolates route params into the current path. |
| `query` | `Record<string, unknown>` | Appends search params to the initial URL. |
| `routeOverrides` | `Partial<Record<string, RouteOverrideOptions>>` | Per-route overrides keyed by route ID, applied to the story's route and root route. Use `'__root__'` to target the root route. Each entry can override `loader`, `beforeLoad`, `validateSearch`, `loaderDeps`, and `context`. |
| `context` | `Record<string, unknown>` | Router context values injected into the story router. |

If `route` is supplied as a plain object, it may also include TanStack route options such as `context`, `head`, `search`, and `parseParams`.

### Modules

The package exports these additional modules:

* `@storybook/tanstack-react/node` for `defineMain()` and `StorybookConfig`
* `@storybook/tanstack-react/preview` for the preview annotations added by the framework
* `@storybook/tanstack-react/react-router` for TanStack Router-compatible mocks
* `@storybook/tanstack-react/start` for TanStack Start-compatible mocks, including a mocked `createServerFn()` implementation
