Problem statement, revisited

From the pitch: "as a developer using an agent to change UI, I want Storybook to point me to the stories most likely to matter."

What the pitch got right

  • Review fatigue from "show every affected story" is real — the worst observed cascade is 1,236 stories (75% of the dogfood) on a single theming-barrel edit.
  • The current change-detection output is useful — it's never wrong about a story being affected, just often noisy.
  • The signal IS imperfect in known ways: CSS-blind, regex-aliased imports opaque, type-only files invisible.

What we learned that refines the problem framing

Confirmed

  • The cascade case is real (21% of files have >100 importers).
  • The cascade case is also the minority case (79% of files have ≤100 importers; 67.9% have ≤10).
  • Inside a cascade, the agent's job is consolidation, not selection. 1,025 stories → 6 readable clusters.

Reframed

  • The pitch describes the agent as a shortlist filter ("pick K from N"). We now believe the agent should be a categorizer ("group all N into K clusters and explain each"). This avoids the false-negative cliff inherent to filtering.
  • 70% of edits resolve to a single modified story. A deterministic "lead with the one modified card" UX already wins for the dominant case — agent or no agent.

Pitch deliverables — status

D1. Agent selection workflow (prompt / skill / context)
working prototype · signature prompt with deterministic expansion

What exists: scripts/eval/inner-loop/prompts/categoriser-signature.md + lib/expand-signatures.ts. Catches every story, never hangs, completes in 14–18 s at any cascade size.

What's still open: the iteration-2 design we now recommend is simpler still — generate cluster content deterministically (namespace + shared-files), use the LLM only to merge over-segmented clusters and write rationales. That collapses output to a few hundred tokens and latency to seconds.

D2. Mechanism for opening a dynamically selected story set
designed, not built · review-slug URL + MCP tool

Recommendation: two-channel — and the URL form is NOT raw story IDs in query params (a 25-story cascade would blow URL length limits):

  • Server-side review store + short slug. A POST endpoint accepts a story-set or cluster-set and returns a short ID; the shareable URL is /review/<uid>. Alternatively, since cluster signatures are already O(K) and short by design, the agent can also POST signatures and let the client re-expand them — cheaper than a server store for the in-session case.
  • MCP tool (open_review_page) for live agent flows that already have a channel into the addon.

Open: the MCP tool surface needs cross-repo work with storybookjs/mcp. Three tools: get_change_context, apply_review_status, open_review_page. Estimated 2–3 days incl. PR review.

D3. Focused review page
design only · iframe pool required

Architecture is agreed: before/after iframes, session-pinned baseline at merge-base (not git HEAD), agent-recommended status taxonomy slotted next to modified/affected.

Hard constraint: iframe pooling is iteration-1 work, not optional. Each iframe costs ~8 MB browser memory; the 75% cascade in compare mode would crash a tab without a pool. Confirmed live.

D4. User evaluation vs current changed-stories baseline
methodology drafted, no real users yet

The eval harness scores recall/precision/cluster-purity against the deterministic ground-truth, but the user-facing evaluation (preference, time-to-first-finding, trust calibration) has not happened. Page 4 of the questions docs has a methodology draft (within-subjects design, ≥6 participants, counterbalanced order). This is the largest unknown in the project.

The 6 research questions from the pitch

Q1. Can the agent produce a useful shortlist from the diff + metadata + change-detection output? What context does it need?
answered

Yes, with two caveats:

  1. "Shortlist" should be "categorization." Filtering is a false-negative cliff; categorising every story into a small number of named clusters has no hidden stories.
  2. The agent must also receive the raw git diff, not only the change-detection statuses. CSS, type-only files, and regex-aliased barrels are systematically invisible to change-detection.

On 8 real-commit replays plus 5 synthetic edits, signature-prompt clusters score recall = 1.0, precision = 1.0. Cluster rationales hand-labelled at 95.2% correct.

Q2. URL params vs MCP for opening a dynamic story set?
answered — both, but URL means slug-not-IDs

Two channels, neither is "stuff every story ID into the query string." A 25-story cascade alone would blow common URL-length limits.

  • Server-side review store + short slug (/review/<uid>) for cold-share / paste-this-link flows.
  • POST cluster signatures (O(K), short by construction) and re-expand client-side — cheaper than a server store for the in-session case, and reuses the signature shape the eval already produces.
  • MCP tool (open_review_page) for live agent flows.

All three can coexist. Building the slug path first is the simpler thing and unblocks the deterministic iteration-1 without depending on the MCP cross-repo work landing.

Q3. What's the right review-page UI?
defaults proposed; user sessions still required to validate

Proposed defaults (from a separate team conversation that converged on the same instincts as the eval):

  • Single-up with baseline ↔ latest toggle, not side-by-side. Page-level stories and viewport-specific components break in any tiny iframe; giving each story the full preview width avoids the worst layout failures and side-steps the "small mobile-viewport rendered in 50%-width frame" trap.
  • Cluster-organised navigation — sections headed by the agent's cluster rationale, story cards inside. Same "zoom levels" intuition the team independently reached (button → header → page) which the categoriser output already supports.
  • Lead with one card when |modified|=1, grid when ties. 70% of edits resolve to a single modified story — the dominant case warrants its own layout, not a degenerate grid-of-one.

Still open and needs user-session input:

  • How aggressively to collapse the "zoom out" affected list — start expanded, collapsed, or smart-collapse beyond N?
  • Where does cluster rationale text appear? Headline, tooltip, expandable detail?
  • Detail/full-screen view for complex components — Page 4 added this as Q.
  • Comment/leave-a-note affordances (raised in team conversation) — iteration-1 or 2?
Q4. Core Storybook feature or addon?
leaning addon

The change-detection engine is already a Storybook core (code/core/src/core-server/change-detection/). The review page currently lives in addon-before-after; the agent layer could live there too. Recommendation: addon for iteration 1, lift to core in 11 if traction.

Q5. How do we recruit unbiased evaluation users?
open — needs DX + Design

Page 4 recommends within-subjects (~6 users), counterbalanced order, real diffs from their own repo, with the agent flow and the deterministic-only flow. Open: recruitment channel (Chromatic users? Storybook Discord power users? both?), incentives, scheduling cadence.

Q6. Can we package this info for Chromatic as a UI-review input signal?
conceptually yes, scope-deferred

The categorizer output ({ clusterId, rationale, representative, stories[] }) is itself a perfectly fine input shape for an external consumer like Chromatic. Iteration-1 should serialise it cleanly so it could be POSTed forward later; actual Chromatic plumbing is a separate project.

Barrel-share — a heuristic that doesn't work

Hypothesised: "exclude barrels from cascade" would cut the bulk of noise. Measured: barrels (*/index.ts(x)) account for only 7.3% of cascade edges; top-10 barrels are 3.4%. The cascade is structural fan-in of non-barrel shared code (manager-api singletons, theme providers, preview-api modules) — depth-based pruning is the only deterministic lever that would matter.

CSS-blast — the known gap, with a deterministic workaround

Change-detection is CSS-blind by design (it walks JS/TS imports). Prototype workaround in lib/css-blast-radius.ts: walk the reverse index, find JS/TS siblings of each .css file, union their importing stories. Verified live: pseudo-states/src/stories/*.css edit synthesises 9 stories vs change-detection's 0. High-recall, low-precision (every CSS file in a directory produces the same set; refinement = filename matching).

Findings — the agent layer is operationally viable

Signature prompt unlocked the cascade scale

The original "enumerate every story per cluster" prompt hangs at >600 stories — the model generates 30K+ output tokens that never stream completely. The signature variant asks for cluster patterns (prefix / regex / explicit IDs) and a deterministic expander fills them. Output stays under 1K tokens regardless of cascade size.

ScenarioCascadeDurationCostRecallPrecisionCluster count
small (Sidebar edit)11612 s$0.0631.01.05–6
medium (Button edit)1,02515 s$0.1471.01.06
large (theming barrel)1,23618 s$0.1761.01.07

Variance is well-bounded

6 repeated runs of the small scenario on Sonnet/signature/low: recall 1.0 stable, precision 1.0 stable, cluster count 5–6, pairwise Jaccard 0.707. Cost drops 4× on runs 2+ thanks to prompt-cache hits.

Haiku works as a primary mode

Counter to Page 4's original "downshift on Haiku" recommendation: claude-haiku-4-5 with signature/low completes all three scenarios with recall=1, precision=1, purity comparable to Sonnet. 4–5× slower (47–80s vs 12–18s) but ~30% cheaper at cascade scale. Haiku is the cost-sensitive primary, not a defensive downshift.

Real-commit replay confirms synthetic findings

8 real dogfood commits replayed via reverse-patch trick (git apply -R). Recall=1.0 and precision=1.0 across every replay; cascades from 12 to 1,025 stories; cost $0.015–$0.151 per commit. Cluster purity is bimodal — 1.0 when the cascade fits in one namespace, ~0.15 when it crosses namespaces (real cross-namespace cascades, not agent failure).

Cluster rationales are trustworthy

Hand-labelled all 21 cluster rationales across the 5 successful real-commit replays: 20 correct (95.2%), 1 partial, 0 wrong. No hallucinated diff details across the sample. Page 4 set 80% as the threshold for "show rationale text directly in the UI" — comfortably cleared. Iteration-1 should NOT add a "model-generated" warning label.

Failure modes degrade gracefully

9 of 12 failure paths (parse error, empty clusters, invalid regex signature, missing catch-all, shadowed cluster, hallucinated IDs, duplicates, empty ground-truth, phantom IDs) exercised directly — all degrade gracefully with sensible scores. 3 documented network/SDK paths (Storybook unreachable, rate-limit, hang) — the only unrecovered path is the SDK hang on enumerate-prompt at cascade, already fixed by the signature variant. Production iteration-1 still needs a wall-clock timeout (~60 s) with deterministic-baseline fallback as the bare-minimum acceptance criterion.

Findings — cost & latency are not the problem

The "expensive AI" worry from the original transcript does not materialise empirically.

Token cost

  • Input: 2.7K (typical) → 22K (cascade) → 63K (worst measured). Categoriser framing keeps us well under the 200K context.
  • Output: signature prompt holds output under 1K tokens regardless of cascade.
  • The "expensive AI" framing was about the filter approach blowing context above 611 stories. Reframing to categoriser fixed it.

Dollar cost per invocation

  • Sonnet/signature/low: $0.06–$0.18 across all scales.
  • Haiku/signature/low: $0.065–$0.13.
  • Real-commit replay range: $0.015 (tiny diff) → $0.15 (1K-cascade).
  • Even a heavy user (50 reviews/day) is under $10/day on Sonnet — cost UI is not needed for iteration 1.

Latency

  • Sonnet at cascade: 14–18 s end-to-end (10–14s API).
  • Haiku at cascade: 47–80 s.
  • First user-visible feedback is the deterministic modified list which exists before any agent call — the agent layer enriches, doesn't gate the page render.

What's clear vs what's not

Clear & settled

  • No architectural blocker — every tier-1 risk resolved positively.
  • Agent should categorize, not filter.
  • Signature-prompt design is the canonical agent shape.
  • Deterministic baseline already handles 70% of edits well; iteration-1 should ship without an agent.
  • Agent input must include raw git diff alongside change-detection status.
  • Session-pinned baseline at merge-base, not git HEAD.
  • Iframe pool is iteration-1 work, not optional.
  • Cost & latency are not blockers; cost UI not needed in iteration 1.
  • Rationales are trustworthy enough to show directly.
  • Failure paths degrade gracefully (with a wall-clock timeout added).
  • experimental_getStatusStore('storybook/agent-review') is the right write surface; round-trips intact through UniversalStore.

Not yet clear (= what to discuss)

  • With-baseline or walk-only for iteration 1? Pitch literally says "before and after states in one place" but the walk-only fork drops the riskiest engineering (iframe pool) and matches the 70%-single-modified data.
  • What does a real user-session evaluation look like? Recruitment, incentives, the diffs they bring, success criteria.
  • What recall threshold ships? Pre-commit to ≥95% now (with Chromatic-as-backstop framing). Measured 100% empirically but variance is non-zero.
  • Is iteration-1 truly deterministic-only? Or do we ship the signature-prompt agent from day one and switch on it for high-fan-in cascades?
  • Core feature or addon? Leaning addon for iteration-1; lift if it earns it.
  • Detail/full-screen story view UX — modal? full-route? lazy-detect dense stories?
  • Cluster-acceptance UX — once the agent ships, can the user accept/reject/edit clusters? Persisted as status? Iteration-2 problem but informs the iteration-1 page shell.
  • Chromatic integration shape — POST forward, separate MCP, both?
  • Storybook 10.5 release branch cuts at end of week 6. No buffer; what scope are we committing to?

Pre-kickoff decisions the team needs to make

From page 0. These are the five questions to answer before week 1.

  1. Two tracks or one project? Recommend: two tracks. Review page = engineering; agent shortlist = research. Separate success bars. Engineering is shippable without the research landing.
  2. Iteration-1 deliverable? Recommend: review page + deterministic MCP tool, no agent. Agent layer is iteration-2 if user sessions ask for it. The 70%-single-modified data backs this; we'd be shipping value to the dominant case immediately.
  3. Recall threshold to ship? Recommend: pick now, not after measurement. ≥95% with explicit "Chromatic catches the rest" framing is realistic and honest. Measured 100% but variance is non-zero.
  4. Eval-time VRT — the "no local VRT" no-go is product-facing; internal-only VRT for hand-labelling ground truth is allowed. Confirm this is the team's read.
  5. Baseline strategy — session-pinned at merge-base, not git HEAD. Required for agentic loops (agent commits drift HEAD; reviewers want to see their own session's changes vs the branch start, not vs the last agent commit).

Recommended iteration-1 spec

Concrete shape we'd commit to in the kickoff. Everything here is buildable inside the 10.5 release branch window. But — there are two credible iteration-1 forks the team needs to pick between before week 1.

The fork: with-baseline vs walk-only

Fork A — with-baseline (the pitch's D3)

Ship before/after for the modified set, single-up with a baseline ↔ latest toggle (not side-by-side). Requires session-pinned baseline + iframe pool with eviction.

Pros: matches the pitch's literal deliverable; closest analogue to Chromatic's UI for users who already use it.

Cons: the iframe pool is the hardest single piece of iteration-1; week-six release branch leaves no buffer.

Fork B — walk-only (raised in team conversation)

No before/after at all. Instead: a cluster-organised story walk — side-scroll or keyboard-next through the agent-determined subset, agent rationale as section header per cluster, only the latest state shown. Optionally a "leave a comment" affordance per story.

Pros: drops the iframe pool from iteration-1's critical path; works with the deterministic-only baseline; ships faster and lower-risk.

Cons: deviates from the pitch's D3 ("before and after states in one place"). The team has to be honest that the user is reviewing against their mental model, not a rendered baseline.

Recommendation: walk-only as iteration 1, with-baseline as iteration 1.5 once iframe pool work has a stable design. Three reasons: (a) the 70%-single-modified statistic means most edits don't need a visual diff to be useful; (b) the no-VRT constraint plus viewport/theme/play-function uncertainties means baseline rendering is the most failure-prone surface; (c) shipping a leaner v1 keeps the door open to learn what users actually do before committing engineering to the harder piece.

What ships either way

  • Review-page route with a short /review/<uid> URL backed by a server-side review store.
  • Cluster-organised navigation: sections headed by rationale, story cards inside, lead with one card when |modified|=1.
  • "Zoom out" panel for the affected set, default-collapsed beyond N stories.
  • get_change_context MCP tool returning the payload shape already serialised in the eval ({ changedFiles, diffSummary, modified, affected, new, cssAffected, projectShape }).
  • apply_review_status MCP tool writing to experimental_getStatusStore('storybook/agent-review').
  • open_review_page MCP tool — a channel event the addon listens for.
  • "Chromatic catches the rest" framing in the UI, regardless of any actual Chromatic plumbing.
  • Wall-clock timeout (60s) around any future agent call, with graceful fallback to the deterministic baseline.
  • Telemetry from day one: cascade size, time-to-first-finding, zoom-out frequency.

Conditional on Fork A only

  • Iframe pool with eviction (the hard piece).
  • Session-pinned baseline at merge-base (mechanism already exists in addon-before-after).
  • Optional: storybook-static folder as baseline if present (cheap to add, opt-in by file-presence detection, raised in team conversation as iteration 1.5).

What's not in iteration 1 either way

  • The agent layer itself (iteration-2 hook only).
  • Cluster acceptance UX (iteration 2 if categoriser ships).
  • Cost UI (cost is not a problem).
  • Local production VRT (per pitch no-go).
  • Webpack-based builder support.

What's the agent's role when it ships in iteration 2?

Based on the deterministic-baselines experiment: the LLM's actual contribution is consolidation, not clustering. Namespace baseline produces 1–118 clusters; the LLM produces 2–7. The simpler iteration-2 design that emerges:

  1. Generate cluster content deterministically (namespace OR shared-changed-file set).
  2. Use the LLM only to (a) merge over-segmented namespace clusters into UX-usable groupings, (b) write a rationale per surviving cluster.
  3. Output drops to ~5–10 short rationales (~500 tokens); latency a few seconds; variance near-zero.
